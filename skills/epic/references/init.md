# epic init

First-time repository setup for the epic workflow system.

## When to Run

- First time using the epic skill in a new repo
- After cloning a repo that doesn't have `.ai-queue/` set up
- To reconfigure project management integration

## Procedure

### 1. Auto-detect owner and repo

```bash
git remote get-url origin
```

Parse the URL to extract `OWNER` and `REPO`:
- `https://github.com/OWNER/REPO.git` → extract directly
- `git@github.com:OWNER/REPO.git` → extract directly
- Strip any trailing `.git` suffix

### 2. Check for existing setup

If `.ai-epics/docs/project-setup.md` already exists, print its contents and ask: **"Project already initialized. Re-run init?"** On "no", abort. On "yes", continue (overwrite).

### 3. Create directory structure

```bash
mkdir -p .ai-queue/_completed
mkdir -p .ai-epics/docs
mkdir -p .ai-epics/roadmaps
mkdir -p .ai-epics/archive
```

> **Note:** `.ai-queue/` is gitignored (per-engineer ephemeral work). `.ai-epics/` is tracked in git (shared project artifacts: docs, roadmaps, archive).

### 3.5. Create feature flag module

Create the flag directory and seed the TypeScript flag registry:

```bash
mkdir -p config
```

Write `config/flags.ts` from the template in `references/feature-flags.md` (the empty registry version). If this file already exists (project already uses feature flags), skip this step.

### 4. PM tool selection

Ask the following questions interactively:

1. **"Which project management tool does this project use?"**
   - GitHub Issues (default)
   - Linear
   - Jira
   - None

2. **"Do you own epics? (Can you create and close epic-level tickets?)"**
   - Yes → owner mode
   - No → contributor mode

   *Explain:* "In contributor mode, I'll ask you for existing issue IDs instead of creating new ones, and I won't close issues when work completes."

3. **Tool-specific questions:**

   | Tool | Questions |
   |------|-----------|
   | GitHub | "Project board number? (optional, for board status tracking)" |
   | Linear | "Linear team key? (e.g., ENG)" / "Is the Linear MCP server available?" |
   | Jira | "Jira project key? (e.g., PROJ)" / "Jira base URL? (e.g., https://myorg.atlassian.net)" |
   | None | No additional questions |

4. **"Notification topic for push alerts? (ntfy topic, optional)"**
   - If provided, configure ntfy
   - If skipped, fall back to desktop notifications

### 5. Run `INIT_LABELS`

Run the `INIT_LABELS` action from the selected PM adapter (`references/pm/{tool}.md`).

- If tool is `none`, skip this step.

### 6. Run `INIT_BOARD`

Run the `INIT_BOARD` action from the selected PM adapter (`references/pm/{tool}.md`).

- If tool is `none`, skip this step.
- Store any board identifiers returned by the adapter.

### 7. Generate `project-setup.md`

Write `.ai-epics/docs/project-setup.md` and commit it:

```markdown
---
owner: OWNER
repo: REPO
flags_dir: config
created: YYYY-MM-DD
---

# Project Setup

Generated by `epic init`. Provides project-specific values for the epic skill.

- **Repository:** OWNER/REPO
- **Feature flags:** config/flags.ts

## Project Management

- **Tool:** {selected-tool}
- **Ownership:** {owner|contributor}
{adapter-specific fields from the selected PM tool}
```

The adapter-specific fields depend on the tool selected in step 4. See each adapter's Config Fields section in `references/pm/{tool}.md`.

### 8. Run `POST_INIT`

Run the `POST_INIT` action from the selected PM adapter (if the adapter defines one).

- GitHub adapter: installs the post-merge cleanup workflow (`.github/workflows/epic-cleanup.yml`)
- Other adapters: may have no post-init steps

### 9. Print summary

```
epic init complete.

Owner:      OWNER
Repo:       REPO
PM Tool:    {tool} ({ownership} mode)
Board:      #{number} (or "not configured")
Labels:     epic, stage, iteration, task (or "skipped")
Flags:      config/flags.ts
Notifications: {ntfy topic} (or "desktop only")

Setup file: .ai-epics/docs/project-setup.md
```

## Notes

- This command is idempotent. Running it again overwrites `project-setup.md`.
- Labels use `--force` or equivalent for safe re-runs.
- `.ai-queue/` is gitignored (per-engineer ephemeral task queue). `.ai-epics/` is committed to the repo for team visibility.
